# FastAPI 后端项目重构任务书

## 🎯 重构目标 (Context & Goal)

本项目是一个基于 FastAPI 的音视频处理后端（包含 FFmpeg 任务调度、WebSocket 进度推送、用户鉴权等）。当前代码功能完善，但随着业务增长，部分单体文件（如 `schemas.py`、`crud.py`）过于庞大，且 API 路由层（Controllers）中耦合了过多底层业务逻辑。

本次重构的**核心目标**是：

1. 按领域（Domain）拆分庞大的模块（Schemas, CRUD）。
2. 将核心业务逻辑从 API Router 层剥离到 Service 层。
3. 统一文件路径处理规范（全面拥抱 `pathlib`）。
4. 增强数据库会话管理的健壮性。

**⚠️ 约束条件：**

- 保持现有 API 接口的 Request/Response 结构完全不变。
- 保持现有数据库表结构和核心业务逻辑（特别是 FFmpeg 硬件加速逻辑）不变。
- 每次完成一个 Phase 后，请确保所有引用（Imports）都被正确更新。

## 🛠️ 任务拆分 (Task Breakdown)

### Phase 1: Pydantic Schemas 领域拆分

当前 `app/schemas/schemas.py` 文件包含了所有的 Pydantic 模型，需要将其按实体拆分。

- [ ] 1. 创建 `app/schemas/user.py`：移入 `UserBase`, `UserCreate`, `User`, `Token`, `TokenData` 等模型。
- [ ] 2. 创建 `app/schemas/file.py`：移入 `FileBase`, `FileCreate`, `File`, `FileStreamInfo`, `FileFormatInfo`, `FileInfoResponse`, `FileResponseInner`, `FileResponseForFrontend` 等模型。
- [ ] 3. 创建 `app/schemas/task.py`：移入 `TaskBase`, `TaskCreate`, `Task`, `ProcessPayload`, `Resolution` 等模型。
- [ ] 4. 创建 `app/schemas/system.py`：移入 `APIResponse`, `SystemCapabilities`。
- [ ] 5. 修改 `app/schemas/__init__.py`：从上述拆分出的模块中导入所有模型，并使用 `__all__` 暴露，以保证原有 `from app.schemas import schemas` 的兼容性，或者全局替换各文件中的引入方式（推荐直接更新全局引用，如 `from app.schemas.user import User`）。
- [ ] 6. 删除原 `app/schemas/schemas.py` 文件。

### Phase 2: CRUD 领域拆分

当前 `app/crud/crud.py` 文件包含了所有数据库操作，需进行拆分。

- [ ] 1. 创建 `app/crud/crud_user.py`：移入 `get_user_by_username`, `create_user`。
- [ ] 2. 创建 `app/crud/crud_file.py`：移入 `create_user_file`, `get_user_files`, `get_file_by_id`, `update_file_status`, `delete_file`。
- [ ] 3. 创建 `app/crud/crud_task.py`：移入 `get_user_tasks`, `get_task`, `create_task`, `update_task`, `delete_task`。
- [ ] 4. 修改 `app/crud/__init__.py`：将所有拆分出来的方法统一导入并导出，以便外部通过 `from app.crud import crud` 调用时不报错（例如 `from .crud_user import *` 等），或全局重构引入路径。
- [ ] 5. 删除原 `app/crud/crud.py` 文件。

### Phase 3: Controller 业务逻辑下沉到 Service 层

将 `app/api/` 下面的路由处理函数中过长的底层逻辑抽离到独立的 Service 脚本中。

- [ ] 1. **FFmpeg 命令构建抽离**：
  - 在 `app/services/` 创建 `ffmpeg_builder.py`。
  - 将 `app/api/process.py` 中的 `construct_ffmpeg_command` 函数移入该文件。
  - 更新 `app/api/process.py` 中的 import。
- [ ] 2. **FFprobe 逻辑抽离**：
  - 在 `app/services/` 创建 `ffprobe_runner.py`。
  - 将 `app/api/download.py` 中的 `run_ffprobe_sync` 移入。
  - 可选：在 `ffprobe_runner.py` 中将原先裸露的 subprocess 异常转化为自定义业务异常。
- [ ] 3. **文件校验逻辑抽离**：
  - 在 `app/services/` 创建 `file_validator.py`。
  - 将 `app/api/upload.py` 中的 `validate_file_signature` 和 `validate_file_type` 移入。

### Phase 4: 统一 Pathlib 路径处理

代码中混用了 `os.path` 和 `pathlib.Path`，请统一为面向对象的 `pathlib.Path`。

- [ ] 1. 扫描 `app/api/upload.py`：将 `os.path.join`, `os.path.normpath` 替换为 `Path()` 对象操作（如 `file_location = user_upload_directory / unique_filename`）。
- [ ] 2. 扫描 `app/api/process.py`：将涉及 `input_path`, `final_output_path`, `temp_output_path` 的 `os.path` 操作重构为 `Path`。
- [ ] 3. 扫描 `app/crud/crud_file.py` (原 `crud.py`) 中的 `delete_file`：使用 `Path(path_to_delete).unlink(missing_ok=True)` 替代 `os.remove`。

### Phase 5: 优化独立线程/异步后台任务的数据库会话管理

部分脱离 FastAPI 依赖注入（`Depends(get_db)`）的方法，手动创建了 DB Session 但缺乏上下文管理，存在连接泄露风险。

- [ ] 1. 检查 `app/api/download.py` 中的 `get_temp_download_link`：
  ```python
  # 将现有的 try...finally 替换为更安全的 contextmanager (如果适用)
  # 或确保发生 HTTPException 时 db.close() 一定会执行。
  ```
- [ ] 2. 检查 `app/services/ffmpeg_runner.py` 中的 `run_ffmpeg_process`：
  - 目前使用了裸露的 `db = SessionLocal()` 和 `db.close()`。
  - 重构建议：在 `app/core/database.py` 中新增一个 `@contextmanager`：
    ```python
    @contextmanager
    def get_db_session():
        db = SessionLocal()
        try:
            yield db
        finally:
            db.close()
    ```
  - 将 `run_ffmpeg_process` 内部重构为使用 `with get_db_session() as db:` 的安全模式。

### Phase 6: 全局异常处理 (Global Exceptions)

消除 API 路由中重复的 try-except HTTPException。

- [ ] 1. 新建 `app/core/exceptions.py`。
- [ ] 2. 定义几个常用业务异常类，例如：
  - `class FileProcessingError(Exception): pass`
  - `class ResourceNotFoundError(Exception): pass`
- [ ] 3. (可选) 修改 `app/main.py`，注册 `@app.exception_handler`，将这些业务异常统一转换为 FastAPI 的 `JSONResponse(status_code=...)` 输出。
